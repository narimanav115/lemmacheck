# LemmaCheck — Документация программного обеспечения

## 1. Назначение программы

**LemmaCheck** — десктопное приложение для полнотекстового поиска по документам с применением лемматизации. Программа позволяет загружать коллекцию документов различных форматов (DOCX, DOC, PDF, TXT, XLSX, XLS), автоматически индексировать их содержимое и выполнять поиск с учётом морфологических форм слов на трёх языках: **русском**, **английском** и **казахском**.

Ключевое отличие от обычного текстового поиска: при запросе «бала» программа находит также все словоформы — «балалар», «баламен», «баласы» и т.д. — благодаря приведению слов к начальной форме (лемме или основе) на этапе индексации.

---

## 2. Используемые методологии

### 2.1. Лемматизация и стемминг

Программа применяет три различных подхода к морфологическому анализу в зависимости от языка текста:

#### Русский язык — словарная лемматизация (pymorphy3)

Для русского языка используется библиотека **pymorphy3** — морфологический анализатор, основанный на словарях OpenCorpora. Метод работает следующим образом:
- Каждое слово разбирается морфологическим анализатором
- Из набора возможных разборов выбирается наиболее вероятный (первый по списку)
- Извлекается нормальная форма (`normal_form`) — лемма
- Примеры: «учебного» → «учебный», «планам» → «план», «университетов» → «университет»

Данный подход обеспечивает высокую точность, поскольку опирается на лингвистический словарь с грамматической разметкой.

#### Английский язык — словарная лемматизация (NLTK WordNet)

Для английского языка используется **WordNetLemmatizer** из библиотеки NLTK с предварительным определением части речи через POS-тегирование:
1. Слово подвергается POS-тегированию (`nltk.pos_tag`) для определения части речи (существительное, глагол, прилагательное, наречие)
2. Тег части речи передаётся лемматизатору WordNet
3. Лемматизатор возвращает базовую форму слова с учётом его синтаксической роли
- Примеры: «studies» → «study» (сущ.), «studying» → «study» (гл.), «better» → «good» (прил.)

#### Казахский язык — суффиксный стемминг (правилоориентированный)

Для казахского языка разработан **собственный суффиксный стеммер**, учитывающий агглютинативную природу казахского языка. Казахский язык относится к тюркским языкам и характеризуется последовательным добавлением суффиксов к основе слова.

Стеммер оперирует следующими категориями суффиксов:

| Категория | Суффиксы | Пример |
|---|---|---|
| Множественное число | -лар, -лер, -дар, -дер, -тар, -тер | балалар → бала |
| Падежные окончания | -ның/-нің (род.), -ға/-ге/-қа/-ке (дат.), -ды/-ді/-ты/-ті (вин.), -да/-де/-та/-те (мест.), -дан/-ден/-тан/-тен/-нан/-нен (исх.), -мен/-бен/-пен (тв.) | мектептен → мектеп |
| Притяжательные | -ым/-ім, -ың/-ің, -сы/-сі, -мыз/-міз, -ңыз/-ңіз, -лары/-лері | баласы → бала |
| Глагольные | -ған/-ген/-қан/-кен (прич. пр. вр.), -атын/-етін, -ушы/-уші, -лық/-лік, -шы/-ші, -сыз/-сіз | білімдер → білім |

Алгоритм стемминга:
1. Слово приводится к нижнему регистру
2. Суффиксы упорядочиваются по длине (от длинного к короткому) для жадного сопоставления
3. Выполняется до 3 итераций отсечения суффиксов (казахский допускает цепочки суффиксов: «балаларымыздан» → «бала» + «лар» + «ымыз» + «дан»)
4. На каждой итерации проверяется, что остаток слова не короче 2 символов

**Автоматическое определение языка** реализовано на уровне символов:
- Если слово содержит специфичные казахские буквы (Ә, Ғ, Қ, Ң, Ө, Ұ, Ү, Һ, І) — применяется казахский стеммер
- Если включён режим «Казахский язык» (чекбокс в интерфейсе) — **все** кириллические слова обрабатываются казахским стеммером (для слов типа «бала», не содержащих специфичных букв)
- Иные кириллические символы — русский лемматизатор pymorphy3
- Латинские символы — английский лемматизатор NLTK

### 2.2. Инвертированный индекс

Для обеспечения эффективного поиска применяется структура данных **инвертированный индекс** (inverted index) — стандартный подход в информационном поиске (Information Retrieval).

Структура индекса:

```
лемма → {
    документ_1: [позиция_1, позиция_5, позиция_23, ...],
    документ_2: [позиция_3, позиция_17, ...],
    ...
}
```

Каждая лемма отображается на словарь документов, а для каждого документа хранится список позиций (в символах от начала текста), где встречается данная лемма. Такая структура позволяет:
- Мгновенно определять, в каких документах встречается искомое слово
- Выполнять фразовый поиск путём проверки последовательности позиций
- Рассчитывать частотные метрики (TF, DF)

### 2.3. TF-IDF ранжирование

Для оценки релевантности используется классическая модель **TF-IDF** (Term Frequency — Inverse Document Frequency):

$$TF(t, d) = \frac{f(t, d)}{|d|}$$

где $f(t, d)$ — количество вхождений леммы $t$ в документ $d$, $|d|$ — общее число слов в документе.

$$IDF(t) = \ln\left(\frac{N}{DF(t)}\right) + 1$$

где $N$ — общее число документов в коллекции, $DF(t)$ — число документов, содержащих лемму $t$. Добавление единицы предотвращает нулевые веса.

$$TF\text{-}IDF(t, d) = TF(t, d) \times IDF(t)$$

Документы ранжируются по убыванию значения TF-IDF, что позволяет выше располагать документы, где искомые термины наиболее значимы.

### 2.4. Фразовый поиск (Phrase Search)

При многословных запросах (например, «білім беру» или «учебный план») программа выполняет **позиционный фразовый поиск**:

1. Запрос разбивается на слова и каждое слово лемматизируется
2. Определяются документы-кандидаты, содержащие **все** леммы запроса (пересечение множеств документов)
3. Для каждого документа-кандидата проверяется последовательность позиций:
   - Берётся каждая позиция первой леммы запроса
   - Проверяется, что позиция следующей леммы идёт после текущей с расстоянием не более `max_distance` символов (по умолчанию 15)
   - Процедура повторяется для всех лемм запроса
4. Если все леммы найдены в правильной последовательности — фиксируется совпадение

Этот алгоритм гарантирует, что найденные результаты содержат именно фразу, а не разрозненные слова.

### 2.5. Обработка составных слов

Слова, записанные через дефис (например, «научно-технический», «ғылыми-техникалық», «Three-cycle»), индексируются двумя способами:
1. Как целое составное слово (лемматизированное поэлементно: «научно-технический» → «научно-технический»)
2. Как отдельные компоненты («научно» и «технический» → «научно» и «технический»)

Это обеспечивает нахождение составных слов как по полной форме, так и по отдельным частям.

## 3. Количественные метрики

Программа вычисляет следующие частотные метрики для каждого найденного слова/фразы и для результатов в целом:

| Метрика | Формула | Описание |
|---|---|---|
| **Абсолютная частота** | $f$ | Количество вхождений слова/фразы в документ |
| **Относительная частота (%)** | $\frac{f}{N} \times 100\%$ | Доля вхождений от общего числа слов |
| **IP10K** (Instances per 10,000) | $\frac{f}{N} \times 10\,000$ | Нормализованная частота на 10 000 слов |
| **IPM** (Instances per Million) | $\frac{f}{N} \times 1\,000\,000$ | Нормализованная частота на 1 000 000 слов |
| **Число вариаций** | — | Количество уникальных словоформ, сведённых к одной лемме |

где $f$ — число совпадений, $N$ — общее число слов в документе (или в совокупности документов для итоговых метрик).

Нормализованные метрики IP10K и IPM позволяют корректно сравнивать частотность между документами разного объёма.

### 3.1. KWIC-конкорданс (Key Word In Context)

Программа поддерживает отображение результатов в виде **KWIC-таблицы** (конкорданса) — стандартного инструмента корпусной лингвистики. KWIC-формат представляет каждое вхождение ключевого слова в виде строки таблицы с четырьмя столбцами:

| Левый контекст | **Ключевое слово** | Правый контекст | Документ |
|---:|:---:|:---|:---|
| ...учебный процесс в | **университете** | включает лекционные и... | report.docx |
| ...высших учебных заведений и | **университетов** | Республики Казахстан... | policy.pdf |

KWIC-конкорданс строится в фоновом потоке `SearchThread` и передаётся в интерфейс через сигнал `kwic_ready`.

#### Параметры KWIC

- **Тип контекста**: «±5 слов», «±10 слов» или «Предложение»
  - При выборе «±5 слов» или «±10 слов» отображается фиксированное количество слов слева и справа от ключевого слова
  - При выборе «Предложение» контекстом служит всё предложение, содержащее ключевое слово

#### Фильтр контекста

Дополнительное поле «Фильтр контекста» позволяет отфильтровать KWIC-строки по наличию указанного слова в левом или правом контексте. Фильтрация выполняется с учётом лемматизации: если пользователь вводит «учебный», будут показаны строки, в контексте которых встречается любая словоформа данной леммы.

Это позволяет отвечать на вопросы вида: «В каком контексте слово X встречается рядом со словом Y?»

#### Экспорт KWIC

Таблица KWIC может быть скопирована в буфер обмена в формате TSV (Tab-Separated Values) для вставки в Excel или другие приложения.

### 3.2. Экспорт данных

Программа поддерживает экспорт результатов поиска в форматах **CSV** и **XLSX** (Microsoft Excel). Доступны три типа экспорта:

#### Таблица результатов (Results Table)

Экспортирует частотные данные по каждой обнаруженной словоформе:

| Столбец | Описание |
|---|---|
| Лемма | Нормализованная форма запроса |
| Словоформа | Конкретная словоформа, обнаруженная в тексте |
| Документ | Имя файла-источника |
| Абс. частота | Количество вхождений словоформы в документ |
| % | Относительная частота (процент от общего числа слов) |
| IP10K | Нормализованная частота на 10 000 слов |
| IPM | Нормализованная частота на 1 000 000 слов |
| TF-IDF | Значение TF-IDF для данной леммы в документе |

#### Конкорданс (Concordance)

Экспортирует KWIC-таблицу целиком:

| Столбец | Описание |
|---|---|
| Левый контекст | Текст слева от ключевого слова |
| Ключевое слово | Найденное слово/фраза |
| Правый контекст | Текст справа от ключевого слова |
| Документ | Имя файла-источника |
| Позиция | Позиция (в символах) в тексте документа |

#### Сводный отчёт (Summary Report)

Экспортирует обобщённую информацию о поиске: дату и время, поисковый запрос, языковой режим, количество документов и слов в корпусе, количество совпадений, частотные метрики (%, IP10K, IPM), число KWIC-строк и версию программы.

#### Файловые соглашения

Имя файла генерируется автоматически: `lemmaCheck_export_{тип}_{ГГГГММДД_ЧЧММ}.{csv|xlsx}`, где тип — `results`, `concordance` или `summary`.

### 5.2. Многопоточность

Для предотвращения блокировки пользовательского интерфейса при обработке больших объёмов данных используются потоки `QThread`:

- **IndexingThread** — фоновая индексация документов с отображением прогресса
- **SearchThread** — фоновое выполнение поискового запроса и построение KWIC-конкорданса (эмитирует сигналы `result_ready` и `kwic_ready`)
- **ExportThread** — фоновый экспорт данных в CSV или XLSX

Связь между потоком и интерфейсом осуществляется через механизм сигналов и слотов (`pyqtSignal`), что обеспечивает потокобезопасность.

### 5.3. Экспорт данных

Экспорт реализован через диалоговое окно `ExportDialog`, в котором пользователь выбирает:
- **Тип экспорта**: таблица результатов, конкорданс (KWIC), сводный отчёт
- **Формат**: CSV или XLSX

Запись файла происходит в `ExportThread` в фоновом потоке с индикацией прогресса.

### 5.4. Персистентность индекса

Построенный индекс может быть сохранён в формате JSON и загружен при следующем запуске программы, что исключает необходимость повторной индексации. Сохраняются:
- Документы и их метаданные
- Инвертированный индекс со всеми позициями
- Частотные данные (document frequency)
- Настройка языкового режима (флаг «Казахский язык»)

## 7. Пример работы

1. Пользователь загружает набор документов на казахском языке
2. Включает режим «Казахский язык» (чекбокс)
3. Программа индексирует все документы, приводя слова к основам через суффиксный стемминг
4. Пользователь вводит запрос «бала» (ребёнок)
5. Система находит все документы, содержащие словоформы: «бала», «балалар», «баламен», «баласы», «балаларымыздың» и т.д.
6. Результаты выводятся с ранжированием по частоте, контекстной подсветкой совпадений и количественными метриками (IP10K, IPM, %)

## 8. Ограничения и особенности

- **Казахский стеммер** является правилоориентированным (rule-based) и не использует словарь. Это означает возможность пере-стемминга (чрезмерное обрезание) или недо-стемминга (недостаточное обрезание) для некоторых словоформ. Тем не менее, для задач полнотекстового поиска суффиксный стемминг обеспечивает приемлемое качество recall.
- Язык слова определяется по символам, а не по контексту. Слова, записанные стандартной кириллицей без казахских букв (например, «бала»), по умолчанию обрабатываются русским анализатором. Для корректной обработки таких слов необходимо включить режим «Казахский язык».
- POS-тегирование для английского языка выполняется на уровне отдельных слов без учёта контекста предложения, что может приводить к неточному определению части речи.
